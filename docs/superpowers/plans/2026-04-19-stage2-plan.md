# Stage 2 Visual Recreation Plan (Hexo Arknights → Raptor / Swift)

> **Goal:** After Stage 1 completed the core blog responsibilities, Stage 2 focuses on recreating the visual language, atmosphere, and reading experience of `hexo-theme-arknights` in a Raptor-native, Swift-first way.
>
> **Rule of Stage 2:** Rebuild the visual system, not the Hexo implementation model.

---

## 0. Stage 2 Scope Definition

### What Stage 2 should achieve

- Bring the site visually closer to `hexo-theme-arknights`
- Establish a reusable visual system instead of ad-hoc styling
- Improve readability, hierarchy, spacing, framing, and navigation chrome
- Preserve the Stage 1 page/component boundaries
- Keep the implementation maintainable and Swift-native

### What Stage 2 should not do by default

These exist in the Hexo theme, but should **not** be treated as mandatory initial scope:

- comments
- search
- PJAX
- background music
- canvas dust / decorative effects
- custom cursors
- Monaco editor embedding
- Mermaid / MathJax expansion
- encryption
- social integrations

These may be revisited later as **Stage 2B** or **Stage 3**, but they must not derail the first visual pass.

## 2. Stage 2 Architecture Principles

### Principle 1: Keep Stage 1 structure intact

Do not collapse Stage 1’s clean boundaries just to mimic Hexo DOM shape.

Preserve and continue using:

- `MainLayout.swift`
- `SiteHeader.swift`
- `PostListItem.swift`
- `PostMeta.swift`
- `Home.swift`
- `ArticlePage.swift`
- `ArchivePage.swift`
- `TagIndexPage.swift`
- `TagDetailPage.swift`
- `CategoryIndexPage.swift`
- `CategoryDetailPage.swift`

### Principle 2: Migrate visual intent, not file structure

Examples:

- Hexo’s header cluster → your `SiteHeader`
- Hexo’s post card style → your `PostListItem`
- Hexo’s article typography → your `ArticlePage` styling
- Hexo’s archive/tag/category page atmosphere → your existing page components

### Principle 3: Build a token system first

Do not begin by styling page-by-page randomly.

First establish:

- colors
- typography
- spacing
- panel surfaces
- border treatment
- content width
- code block style

### Principle 4: Add ornamental behavior only after the static visual pass succeeds

No motion/detail work until:

- layout feels correct
- typography is readable
- panels and cards are coherent
- article pages feel intentional

## 4. Stage 2A: Establish the Design Token Layer

This is the first real implementation step.

### Goal

Create a minimal but reusable visual vocabulary before styling pages.

### Source inspiration from Hexo

Look at:

- `source/css/_core/color/base.styl`
- `source/css/_core/color/dark.styl`
- `source/css/_core/color/light.styl`
- `source/css/_core/base.styl`
- `source/css/_core/core.styl`
- font assets under `source/font/`

### Extract these token categories

#### Colors

- page background
- panel background
- elevated panel background
- primary text
- secondary text
- accent color
- border color
- code background
- code text
- link color
- muted divider color

#### Typography

- site title font style
- page title size
- section title size
- card title size
- body size
- meta size
- code font

#### Spacing

- page outer margin
- content vertical rhythm
- card internal padding
- section gap
- nav item spacing

#### Shape and framing

- panel border thickness
- corner radius
- shadow or no shadow
- divider behavior

#### Layout width

- page max width
- article reading width
- list page width
- optional side rail width later

### Recommended implementation direction

Create a dedicated style support layer in Swift, for example:

- `Sources/Support/BlogTheme.swift`
- or `Sources/Support/ThemeTokens.swift`

This layer should define named constants or helper APIs for:

- spacing
- widths
- font roles
- color roles
- panel behavior

### Stage 2A checklist

- [ ] Decide dark-first base palette
- [ ] Define named tokens instead of magic values
- [ ] Decide whether custom fonts are used now or deferred
- [ ] Define content width and panel spacing rules
- [ ] Define article typography baseline

## 5. Stage 2B: Rebuild the Global Shell

### Goal

Transform the current plain shell into an arknights-inspired reading frame.

### Source inspiration from Hexo

Look at:

- `layout/includes/layout.pug`
- `layout/includes/header.pug`
- `layout/includes/aside.pug`
- `source/css/_core/header/*.styl`
- `source/css/_core/layout/*.styl`
- `source/css/_core/aside/*.styl`
- `source/img/bg.jpg`
- `source/img/pc-bg.jpg`
- `source/img/bk.jpg`

### Main implementation target

Refine:

- `MainLayout.swift`
- `SiteHeader.swift`

### Visual goals

- stronger outer framing
- a more deliberate header silhouette
- more intentional spacing above and below content
- readable central content region
- optional background image layer or gradient atmosphere
- footer that feels integrated instead of default/plain

### Specific tasks

#### Task B1: Improve `MainLayout`

- introduce controlled page width
- add top/bottom rhythm
- establish a consistent main content frame
- add background or background layer support
- ensure all pages inherit the new shell automatically

#### Task B2: Redesign `SiteHeader`

- make site title/logo area visually distinctive
- improve nav spacing and alignment
- add subtle panel treatment or underline/divider language
- support active-feeling navigation styling if practical

#### Task B3: Decide sidebar strategy

Hexo theme has a stronger aside presence, but Stage 2 should not force it too early.

Choose one:

- **Option 1:** keep a single-column shell for Stage 2A/2B
- **Option 2:** add a light right rail only for article pages later

Recommended: start with Option 1.

### Stage 2B checklist

- [ ] Layout width looks intentional
- [ ] Header no longer feels starter/default
- [ ] Background treatment supports the atmosphere without hurting readability
- [ ] Footer matches the site frame

### Implementation target

Refine:

- `ArticlePage.swift`
- article typography styling
- code block presentation
- previous/next navigation styling

### Specific tasks

#### Task E1: Article header treatment

- stronger title scale
- subtitle styling refinement
- metadata separation from content
- improve top spacing and breathing room

#### Task E2: Article body typography

- line height
- paragraph spacing
- heading hierarchy
- list spacing
- blockquote styling
- inline code / code block readability
- link styling inside article content

#### Task E3: Code and technical content styling

Use Hexo’s code styles only as reference, not as a file to copy.

Decide:

- code background color
- border treatment
- padding
- mono font choice
- overflow behavior

#### Task E4: Previous/Next navigation block

Make article navigation feel like themed secondary cards rather than plain links.

#### Task E5: Optional table of contents decision

Hexo theme has TOC-related styling and aside patterns.

For Stage 2 initial pass choose one:

- **Option A:** no TOC yet
- **Option B:** simple in-page TOC if cheap and clean

Recommended: Option A unless Raptor gives a clean built-in path.

## 9. Stage 2F: Assets and Font Integration

### Goal

Use the Hexo theme assets selectively and deliberately.

### Candidate assets from Hexo

- fonts under `source/font/`
- background images under `source/img/`
- icon assets under `source/icons/`
- favicon

### Rules

- Do not copy every asset blindly
- Do not add decorative assets until the layout works without them
- Use custom fonts only if they improve the feel without hurting readability or performance

### Suggested order

#### Step F1: Decide custom font usage

Potential candidates:

- one display font for site title / headings
- one mono font for code

#### Step F2: Decide background strategy

Potential candidates:

- one subtle full-page background
- one article/page-surface background texture

#### Step F3: Decide icon integration

Use only a small subset if useful.

### Stage 2F checklist

- [ ] Fonts improve the mood and hierarchy
- [ ] Background does not reduce contrast/readability
- [ ] Assets are additive, not compensating for weak layout

## 15. Final Working Rule

When looking at any Hexo file during Stage 2, ask these two questions first:

1. What user-facing behavior or visual idea does this file represent?
2. What is the cleanest Swift/Raptor representation of that idea?

If the answer becomes “copy the original structure,” stop and rethink.

Stage 1 built a functional blog system.

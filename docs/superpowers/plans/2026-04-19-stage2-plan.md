> **Goal:** After Stage 1 completed the core blog responsibilities, Stage 2 focuses on recreating the visual language, atmosphere, and reading experience of `hexo-theme-arknights` in a Raptor-native, Swift-first way.
>
> **Rule of Stage 2:** Rebuild the visual system, not the Hexo implementation model.

> **Updated interpretation after studying Raptor internals:** Stage 2 is not just a styling pass. It is a responsibility-mapping pass across Raptorâ€™s existing layers: `Theme`, `Style`, `PostProcessor`, `PostWidget`, `Layout`, `Page`, and only a thin layer of fallback CSS/JS when necessary.

## 0. Stage 2 Scope Definition

### What Stage 2 should achieve

- Bring the site visually closer to `hexo-theme-arknights`
- Establish a reusable visual system instead of ad-hoc styling
- Improve readability, hierarchy, spacing, framing, and navigation chrome
- Preserve the Stage 1 page/component boundaries
- Keep the implementation maintainable and Swift-native

### What Stage 2 should not do by default

These exist in the Hexo theme, but should **not** be treated as mandatory initial scope:

- comments
- search
- PJAX
- background music
- canvas dust / decorative effects
- custom cursors
- Monaco editor embedding
- Mermaid / MathJax expansion
- encryption
- social integrations

These may be revisited later as **Stage 2B** or **Stage 3**, but they must not derail the first visual pass.

## 2. Stage 2 Architecture Principles

### Principle 1: Keep Stage 1 structure intact

Do not collapse Stage 1â€™s clean boundaries just to mimic Hexo DOM shape.

Preserve and continue using:

- `MainLayout.swift`
- `SiteHeader.swift`
- `PostListItem.swift`
- `PostMeta.swift`
- `Home.swift`
- `ArticlePage.swift`
- `ArchivePage.swift`
- `TagIndexPage.swift`
- `TagDetailPage.swift`
- `CategoryIndexPage.swift`
- `CategoryDetailPage.swift`

### Principle 2: Migrate visual intent, not file structure

Examples:

- Hexoâ€™s header cluster â†’ your `SiteHeader`
- Hexoâ€™s post card style â†’ your `PostListItem`
- Hexoâ€™s article typography â†’ your `ArticlePage` styling
- Hexoâ€™s archive/tag/category page atmosphere â†’ your existing page components

### Principle 3: Start from Raptor's built-in layers, not from CSS files

Raptor already provides meaningful architectural boundaries for a blog system.

Stage 2 should assign responsibilities like this:

- `Theme`
  - global typography system
  - font families
  - font sizes / weights / line heights
  - accent / foreground / background / gradients
  - content width
  - code themes
  - light / dark variants
- `Style`
  - reusable visual components
  - post cards
  - metadata rows
  - taxonomy pills
  - archive group containers
  - future callouts / cards
- `PostProcessor`
  - Markdown preprocessing and enhancement
  - excerpt rules
  - heading cleanup
  - advanced code block processing
  - future Markdown extensions
- `PostWidget`
  - article-embedded rich components
  - note / warning / important blocks
  - repo cards
  - code include blocks
  - demo or reference boxes
- `Layout` / `Page` / `PostPage`
  - page structure and composition
  - shell, routing, hierarchy, page-specific arrangement
- thin CSS / JS fallback
  - only for cases the Raptor layers do not express cleanly

### Principle 4: Theme first, then Style, then Layout/Page polish

Do not begin Stage 2 by spreading visual tweaks across pages.

The preferred order is:

1. `Theme`
   - establish global visual tone
   - typography
   - colors
   - widths
   - code appearance
2. `Style`
   - establish reusable component visuals
3. `Layout` / `Page`
   - refine shell and page composition using the already-defined theme/style system
4. `PostProcessor` / `PostWidget`
   - add content-system enhancements after the baseline visual pass is stable

### Principle 5: Add ornamental behavior only after the static visual pass succeeds

No motion/detail work until:

- theme tone feels correct
- typography is readable
- panels and cards are coherent
- article pages feel intentional
- reusable styles exist for shared components

## 9.5. Future Content-System Layer (PostProcessor + PostWidget)

### Why this matters

Once the visual baseline is stable, the next major upgrade path for the blog is not more page chrome, but better article content capabilities.

### `PostProcessor` should eventually own

- excerpt rules
- heading cleanup (for example first-H1 treatment if needed)
- custom Markdown preprocessing
- advanced code block preprocessing
- future syntax such as admonition-like or callout-like content rules

### `PostWidget` should eventually own

- article-embedded callouts
- notes / warnings / important boxes
- reference cards
- repo cards
- code include widgets
- demo boxes

### Stage 2 implication

These are **not** required to finish the first visual pass, but the Stage 2 architecture should leave room for them instead of forcing all future enhancements into page code.

### Goal

Use the Hexo theme assets selectively and deliberately.

### Candidate assets from Hexo

- fonts under `source/font/`
- background images under `source/img/`
- icon assets under `source/icons/`
- favicon

### Rules

- Do not copy every asset blindly
- Do not add decorative assets until the layout works without them
- Use custom fonts only if they improve the feel without hurting readability or performance

### Suggested order

#### Step F1: Decide custom font usage

Potential candidates:

- one display font for site title / headings
- one mono font for code

#### Step F2: Decide background strategy

Potential candidates:

- one subtle full-page background
- one article/page-surface background texture

#### Step F3: Decide icon integration

Use only a small subset if useful.

### Stage 2F checklist

- [ ] Fonts improve the mood and hierarchy
- [ ] Background does not reduce contrast/readability
- [ ] Assets are additive, not compensating for weak layout

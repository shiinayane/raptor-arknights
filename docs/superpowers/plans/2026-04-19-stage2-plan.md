# Stage 2 Visual Recreation Plan (Hexo Arknights → Raptor / Swift)

> **Goal:** After Stage 1 completed the core blog responsibilities, Stage 2 focuses on recreating the visual language, atmosphere, and reading experience of `hexo-theme-arknights` in a Raptor-native, Swift-first way.
>
> **Rule of Stage 2:** Rebuild the visual system, not the Hexo implementation model.
>
> **Updated interpretation after studying Raptor internals:** Stage 2 is not just a styling pass. It is a responsibility-mapping pass across Raptor’s existing layers: `Theme`, `Style`, `PostProcessor`, `PostWidget`, `Layout`, `Page`, and only a thin layer of fallback CSS/JS when necessary.

**Current implementation correction after deeper Raptor + Hexo analysis:** Stage 2 should not begin from a traditional top-header single-column blog shell. The target shell is a three-region site frame inspired by Hexo Arknights: left navigation, center content, right contextual aside. The first pass may simplify behavior, but the structural model should be aligned with that final direction from the start.

---

## 0. Stage 2 Scope Definition

### What Stage 2 should achieve

- Bring the site visually closer to `hexo-theme-arknights`
- Establish a reusable visual system instead of ad-hoc styling
- Improve readability, hierarchy, spacing, framing, and navigation chrome
- Preserve the Stage 1 page/component boundaries
- Keep the implementation maintainable and Swift-native
- Establish a three-region site shell (navigation / content / aside) without forcing full interaction parity on day one
- Introduce a project-level visual semantics layer for theme-specific typography and component roles that Raptor's built-in theme roles do not cover

### What Stage 2 should not do by default

These exist in the Hexo theme, but should **not** be treated as mandatory initial scope:

- comments
- search
- PJAX
- background music
- canvas dust / decorative effects
- custom cursors
- Monaco editor embedding
- Mermaid / MathJax expansion
- encryption
- social integrations

Also do not treat temporary single-column simplification as the default architectural direction. If behavior must be simplified early, simplify interaction richness first, not the final shell model.

These may be revisited later as **Stage 2B** or **Stage 3**, but they must not derail the first visual pass.

## 2. Stage 2 Architecture Principles

### Principle 1: Keep Stage 1 structure intact

Do not collapse Stage 1’s clean boundaries just to mimic Hexo DOM shape.

Preserve and continue using:

- `MainLayout.swift`
- `SiteHeader.swift`
- `PostListItem.swift`
- `PostMeta.swift`
- `Home.swift`
- `ArticlePage.swift`
- `ArchivePage.swift`
- `TagIndexPage.swift`
- `TagDetailPage.swift`
- `CategoryIndexPage.swift`
- `CategoryDetailPage.swift`

However, preserving Stage 1 boundaries does **not** mean preserving a premature shell interpretation. Existing files should be evolved toward the correct Arknights-style responsibilities rather than kept visually or structurally plain.

### Principle 2: Migrate visual intent, not file structure

Examples:

- Hexo’s header cluster → your `SiteHeader`
- Hexo’s post card style → your `PostListItem`
- Hexo’s article typography → your `ArticlePage` styling
- Hexo’s archive/tag/category page atmosphere → your existing page components

### Principle 3: Start from Raptor's built-in layers, but add a project-level semantics layer where Raptor is intentionally coarse

Raptor already provides meaningful architectural boundaries for a blog system, but its built-in typography/theme roles are intentionally limited. Stage 2 should therefore use **two semantic layers** instead of one:

- `Theme`
  - global typography baseline that Raptor already supports well
  - body / title / code defaults
  - accent / foreground / background / gradients
  - content width baselines
  - code themes
  - light / dark variants
- project-level support layer (for example `BlogTheme`, `Palette`, `Metrics`, `Typography`)
  - theme-specific color tokens beyond Raptor's built-in coarse roles
  - spacing and shell metrics
  - text semantics such as site title, nav label, post meta, taxonomy label, action text
  - reusable mapping from theme-specific semantics to Raptor's built-in typography + direct values where needed
- `Style`
  - reusable visual components
  - post cards
  - metadata rows
  - taxonomy pills
  - archive group containers
  - future callouts / cards
- `PostProcessor`
  - Markdown preprocessing and enhancement
  - excerpt rules
  - heading cleanup
  - advanced code block processing
  - future Markdown extensions
- `PostWidget`
  - article-embedded rich components
  - note / warning / important blocks
  - repo cards
  - code include blocks
  - demo or reference boxes
- `Layout` / `Page` / `PostPage`
  - page structure and composition
  - site shell, routing, hierarchy, page-specific arrangement
  - layout slots for navigation / content / contextual aside
- thin CSS / JS fallback
  - only for cases the Raptor layers do not express cleanly

The key interpretation is:

> Raptor's built-in theme roles are the **base semantic layer**, not the complete theme-language layer.

Stage 2 therefore should not try to force every Hexo visual role into Raptor's fixed typography roles. Instead, it should add a project-owned semantic layer above them.

### Principle 4: Theme baseline first, then project semantics, then shell, then component/page polish

Do not begin Stage 2 by spreading visual tweaks across pages.

The preferred order is:

1. `Theme`
   - establish the coarse global visual baseline
   - typography
   - colors
   - widths
   - code appearance
2. project-level support layer
   - palette
   - metrics
   - theme-specific typography semantics
   - reusable semantic helpers for nav/meta/taxonomy/action text and similar roles
3. `Style`
   - establish reusable component visuals
4. `Layout` / `Page`
   - refine the shell and page composition using the already-defined theme/style/semantic system
5. `PostProcessor` / `PostWidget`
   - add content-system enhancements after the baseline visual pass is stable

### Principle 5: Add ornamental behavior only after the static visual pass succeeds

No motion/detail work until:

- theme tone feels correct
- typography is readable
- panels and cards are coherent
- article pages feel intentional
- reusable styles exist for shared components

## 4. Stage 2A: Establish the Theme Layer

This is the first real implementation step.

### Goal

Create the global visual baseline in Raptor's `Theme` layer before styling individual components or pages. This layer should stay intentionally coarse and should **not** be overloaded with every theme-specific visual role.

### Source inspiration from Hexo

Look at:

- `source/css/_core/color/base.styl`
- `source/css/_core/color/dark.styl`
- `source/css/_core/color/light.styl`
- `source/css/_core/base.styl`
- `source/css/_core/core.styl`
- font assets under `source/font/`

### Extract these theme categories

#### Colors

- page background
- panel background
- elevated panel background
- primary text
- secondary text
- accent color
- border color
- code background
- code text
- link color
- muted divider color

#### Typography

- site title font style
- page title size
- section title size
- card title size
- body size
- meta size
- code font

#### Spacing

- page outer margin
- content vertical rhythm
- card internal padding
- section gap
- nav item spacing

#### Shape and framing

- panel border thickness
- corner radius
- shadow or no shadow
- divider behavior

#### Layout width

- page max width
- article reading width
- list page width
- optional side rail width later

### Recommended implementation direction

Create a dedicated support cluster in Swift, for example:

- `Sources/Support/BlogTheme.swift`
- `Sources/Support/ArknightsPalette.swift`
- `Sources/Support/ArknightsMetrics.swift`
- `Sources/Support/ArknightsTypography.swift`

Responsibilities should be split like this:

- `Theme`
  - body / title / code defaults
  - global light/dark baseline
  - coarse global widths
- `Palette`
  - background / surface / text / accent / border semantics used across the project
- `Metrics`
  - shell widths, spacing, panel padding, framing measurements
- `Typography`
  - project-level text roles such as site title, article title, post meta, taxonomy label, nav label, action text
  - mapping from those roles to Raptor typography + direct values when needed

A small supporting token layer is therefore not optional decoration; it is the correct place to hold the theme-specific language that Raptor's fixed theme roles do not represent cleanly.

### Stage 2A checklist
- [ ] Decide dark-first base palette
- [ ] Define body / title / code font roles in Raptor Theme
- [ ] Define project-level text semantics beyond title1-title6/body/code
- [ ] Define shell metrics (navigation width, aside width, content width, reading width)
- [ ] Define global title scale and text rhythm
- [ ] Decide whether custom fonts are used now or deferred
- [ ] Define article typography baseline
- [ ] Define code block and inline code defaults

### Visual goals

- a deliberate three-region frame: left navigation, center content, right contextual aside
- stronger outer framing
- more intentional shell spacing and region balance
- readable central content region
- optional background image layer or gradient atmosphere
- footer treatment that feels integrated instead of default/plain

### Specific tasks

#### Task B1: Rebuild `MainLayout` as the site shell

- define explicit regions for navigation / content / aside
- allow the aside region to be empty or simplified without changing the shell model
- establish controlled shell widths and gaps
- ensure all pages inherit the new shell automatically
- avoid a temporary top-header single-column model that would later need structural reversal

#### Task B2: Reinterpret `SiteHeader` as the left navigation system

- make site title/logo area visually distinctive
- improve nav spacing and alignment
- add subtle panel treatment or underline/divider language
- support active-feeling navigation styling if practical
- treat this component conceptually as site navigation, even if the file name remains `SiteHeader`

#### Task B3: Introduce contextual aside support

- preserve a right-side region in the shell from the beginning
- allow early versions to keep it simple or partially empty
- plan for page-specific content such as TOC, site stats, archive navigation, or profile/context blocks later
- do not force complete interaction parity before the structural shell is correct

### Stage 2B checklist
- [ ] Shell uses the correct three-region model
- [ ] Navigation no longer feels like a starter/default top bar
- [ ] Central content region is readable and intentionally framed
- [ ] Aside region exists structurally even if simplified at first

## 9.5. Future Content-System Layer (PostProcessor + PostWidget)

### Why this matters

Once the visual baseline is stable, the next major upgrade path for the blog is not more page chrome, but better article content capabilities.

### `PostProcessor` should eventually own

- excerpt rules
- heading cleanup (for example first-H1 treatment if needed)
- custom Markdown preprocessing
- advanced code block preprocessing

### `PostWidget` should eventually own

- article-embedded callouts
- notes / warnings / important boxes
- reference cards
- repo cards
- code include widgets
- demo boxes

### Stage 2 implication

These are **not** required to finish the first visual pass, but the Stage 2 architecture should leave room for them instead of forcing all future enhancements into page code.

### Goal

Use the Hexo theme assets selectively and deliberately.

### Candidate assets from Hexo

- fonts under `source/font/`
- background images under `source/img/`
- icon assets under `source/icons/`
- favicon

### Rules

- Do not copy every asset blindly
- Do not add decorative assets until the layout works without them
- Use custom fonts only if they improve the feel without hurting readability or performance

### Suggested order

#### Step F1: Decide custom font usage

Potential candidates:

- one display font for site title / headings
- one mono font for code

#### Step F2: Decide background strategy

Potential candidates:

- one subtle full-page background

#### Step F3: Decide icon integration

### Stage 2F checklist

## 15. Final Working Rule

When looking at any Hexo file during Stage 2, ask these two questions first:

1. What user-facing behavior or visual idea does this file represent?
2. What is the cleanest Swift/Raptor representation of that idea?

If the answer becomes “copy the original structure,” stop and rethink.

Also ask a third question whenever a visual role appears too specific for Raptor's built-in theme roles:

3. Should this idea live in the project-owned semantic support layer rather than be forced into Raptor's coarse global theme roles?

## 16. Summary

Stage 1 built a functional blog system.
Stage 2 should make it feel like a deliberate, atmospheric, arknights-inspired site without sacrificing the cleaner Raptor architecture.

More importantly, Stage 2 should place responsibilities into the right Raptor layers:

- `Theme` for the coarse global visual baseline
- project-owned support semantics (`Palette`, `Metrics`, `Typography`) for the theme language that exceeds Raptor's built-in role set
- `Style` for reusable component visuals
- `Layout` / `Page` / `PostPage` for shell structure and composition
- future `PostProcessor` / `PostWidget` for richer blog content systems

The correct mindset is:

> **Rebuild the theme language, not the Hexo codebase — let Raptor own the coarse global baseline, let your project own the theme-specific semantics, and let the shell structure match the real Arknights layout model from the beginning.**
